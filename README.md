# ccx

One config source for Claude Code and Codex.

Project-level setup is the preferred path. User-level setup still exists, but it
is much less tested and should be treated as experimental for now.

## Install

```bash
uv tool install ccx
```

## Recommended Project Setup

```bash
cd my-project
ccx init --project

$EDITOR .ccx/AGENTS.md
$EDITOR .ccx/mcp.yaml
$EDITOR .ccx/hooks.yaml

ccx sync
git add .ccx .gitignore .vscode/settings.json
git commit -m "chore: add ccx project config"
```

`ccx init --project` creates the project source folder, updates `.gitignore`,
and writes `.vscode/settings.json` so VS Code hides the files generated by
`ccx sync`.

Teammates only need to clone the repo and run:

```bash
ccx sync
```

## What To Commit

Commit the source config:

```text
.ccx/
|-- AGENTS.md
|-- mcp.yaml
|-- hooks.yaml
|-- skills/
|-- claude/
|   `-- settings.json
`-- codex/
    `-- config.toml
```

Also commit the `.gitignore` block and `.vscode/settings.json` created by
`ccx init --project`.

Do not commit generated targets:

```text
.ccx/.state/
.ccx/backups/
CLAUDE.md
AGENTS.md
.mcp.json
.claude/
.codex/
.agents/
```

## What Goes In `.ccx`

| File or folder | Purpose |
|---|---|
| `.ccx/AGENTS.md` | Shared project instructions for both tools |
| `.ccx/mcp.yaml` | MCP servers |
| `.ccx/hooks.yaml` | Shared hook events |
| `.ccx/skills/<name>/SKILL.md` | Skills shared by Claude Code and Codex |
| `.ccx/claude/` | Claude-only passthrough files |
| `.ccx/codex/` | Codex-only passthrough files |

Example `AGENTS.md`:

```markdown
# Project conventions

- Run tests before committing.
- Prefer pnpm over npm.
- Ask before adding production dependencies.
```

Example `mcp.yaml`:

```yaml
servers:
  context7:
    command: npx
    args: ["-y", "@upstash/context7-mcp"]
  figma:
    url: "https://mcp.figma.com/mcp"
    bearer_token_env_var: FIGMA_OAUTH_TOKEN
```

Example `hooks.yaml`:

```yaml
hooks:
  PreToolUse:
    - matcher: Bash
      hooks:
        - command: "$HOME/.ccx/hooks/pre_bash.py"
          timeout: 30
```

## What `ccx sync` Generates

From project `.ccx/`, `ccx sync` writes native tool config in the repo:

| Generated target | Purpose |
|---|---|
| `CLAUDE.md` | Claude Code include stub |
| `AGENTS.md` | Codex project memory symlink to `.ccx/AGENTS.md` |
| `.mcp.json` | Claude Code project MCP config |
| `.claude/` | Claude Code hooks, settings, and skills link |
| `.codex/` | Codex hooks, config, rules, and skills link |
| `.agents/` | Codex skills link |

Generated files are safe to recreate. If ccx needs to replace a pre-existing
target, it backs it up under `.ccx/backups/` first.

To remove generated targets, run:

```bash
ccx clean
```

`ccx clean` runs the same safety check as `ccx status` first. If anything is
stale, orphaned, or manually edited, it refuses to delete files.

## Tool-Specific Config

Use passthrough folders when a setting does not translate cleanly.

Claude-only settings:

```json
{
  "permissions": {
    "allow": ["Bash(git *)", "Bash(pnpm *)"],
    "deny": ["Bash(rm -rf *)"]
  }
}
```

Save as:

```text
.ccx/claude/settings.json
```

Codex-only config:

```toml
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

Save as:

```text
.ccx/codex/config.toml
```

Codex rules go in:

```text
.ccx/codex/rules/*.rules
```

## User-Level Setup

User-level config is supported but not the recommended path yet.

```bash
ccx init
$EDITOR ~/.ccx/AGENTS.md
$EDITOR ~/.ccx/mcp.yaml
$EDITOR ~/.ccx/hooks.yaml
ccx sync
```

Use this only when you intentionally want personal config across many repos.
Project-level `.ccx/` remains the better-tested workflow.

When both scopes exist:

| Concept | Project + user behavior |
|---|---|
| MCP servers | Project overrides user by server name in project output |
| Hooks | Project hooks append under the same event |
| Memory | Tools load user and project memory through their own hierarchy |
| Skills | User and project skill links point at their own `.ccx/skills` folders |
| Passthrough | User renders to home config; project renders to repo config |

## Not Synced

ccx does not sync tool-owned auto memory. Claude Code and Codex write different
auto-memory formats, and each tool expects to manage its own files.

ccx also does not translate permissions, sandbox rules, subagents, slash
commands, or plugins. Put tool-specific files under `.ccx/claude/` or
`.ccx/codex/`.

## Commands

| Command | Purpose |
|---|---|
| `ccx init --project` | Scaffold `.ccx/`, update `.gitignore`, and add VS Code hides |
| `ccx init` | Scaffold `~/.ccx/` user config |
| `ccx sync` | Render `.ccx/` and/or `~/.ccx/` to native targets |
| `ccx sync --claude` | Render only Claude Code targets |
| `ccx sync --codex` | Render only Codex targets |
| `ccx sync --dry-run` | Print the write plan without changing files |
| `ccx sync --check` | Exit non-zero if targets are stale |
| `ccx sync --force` | Overwrite manual edits to managed files |
| `ccx status` | Show stale or orphaned targets |
| `ccx clean` | Remove generated targets after a clean status check |
| `ccx restore [--list]` | Restore from a backup snapshot |
| `ccx claude [args...]` | Sync, then run `claude` |
| `ccx codex [args...]` | Sync, then run `codex` |

Set `CCX_SKIP_SYNC=1` to skip the automatic sync done by `ccx claude` and
`ccx codex`.
